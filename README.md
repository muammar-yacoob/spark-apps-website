# SparkStripe

Universal Stripe payment backend. Deploy once, add payments to any app, with no Stripe SDK required on the client side.

Live demo: **[sparkstripe.com](https://sparkstripe.com)** &nbsp;|&nbsp; [YouTube Tutorial](https://youtube.com) &nbsp;|&nbsp; [Discord](https://discord.gg/vPTuKEx9Dk)

## What it does

- **Hosted pricing pages** at `/pricing/{app_id}`: drop-in, no frontend work
- **Subscription + one-time payments** via Stripe Checkout
- **Multi-app**: one backend, unlimited apps, each fully isolated
- **Subscription status API**: one REST call from any framework
- **Receipt emails** sent automatically after purchase
- **PDF invoices**: downloadable receipts with full tax breakdown (subtotal, tax rate, total), billing period, and app logo, Stripe Tax compatible
- **Excel revenue reports**: pre-formatted `.xlsx` download with a Summary sheet (total revenue, transaction count) and a Transactions sheet (date, app, customer, plan, amount, currency, USD exchange rate, receipt ref, status), ready for accountants and tax filing
- **Follow-up email campaigns**: segment-based targeting (nearing expiry, recently expired, long expired) with customizable templates, scheduled via QStash
- **Pricing page customization**: feature comparison tables, background animations (gradient, bokeh, bubbles, shooting stars), seasonal promotional badges, trust badges, and title glow effects
- **Multi-currency support**: automatic currency detection and price conversion with exchange rate tracking for revenue normalization
- **Coupon management**: create, validate, and publicly list discount codes per app, linked to Stripe coupon objects
- **Seasonal badges**: time-limited promotional sale badges with discount percentages, displayed on pricing cards and in follow-up emails

### How it works

![Payment flow: Stripe event to SparkStripe backend to client app gating](docs/imgs/Flowchart.png)

Stripe fires a payment event, SparkStripe processes it and updates the database, then your app checks status via polling or receives a signed webhook push. No Stripe SDK needed on the client side.

## Server Setup

### Option A: Automated (recommended)

```bash
bash cli-setup/server-setup/setup.sh
```

Walks you through Vercel login, Neon DB provisioning, Stripe key configuration, and schema push. Auto-generates `AUTH_SECRET` if not set. Requires the [Vercel CLI](https://vercel.com/docs/cli) and optionally the [Stripe CLI](https://docs.stripe.com/stripe-cli).

### Option B: Manual

```bash
bun install
cp .env.example .env.local
# Fill in DATABASE_URL, STRIPE_TEST_KEY, STRIPE_WEBHOOK_SECRET, Google OAuth keys
bun run db:push
bun dev
```

See `.env.example` for all required variables. `AUTH_SECRET` is auto-generated by `setup.sh`, or run `openssl rand -base64 32` manually.

### After deployment

Go to **[Stripe Dashboard, Webhooks](https://dashboard.stripe.com/webhooks)**, Add endpoint:

- **URL:** `https://your-domain.com/api/webhooks/stripe`
- **Events:** `checkout.session.completed`, `customer.subscription.created`, `customer.subscription.updated`, `customer.subscription.deleted`, `charge.refunded`

Copy the signing secret (`whsec_...`) and save it as `STRIPE_WEBHOOK_SECRET` in Vercel.

## Integrate

### Download the integration kit

From the dev dashboard, select your app, go to the **General** tab, and pick your **platform** and **integration mode**:

**Platform:**
- **Next.js**: Uses `NEXT_PUBLIC_` env vars and Next.js fetch revalidation for caching.
- **Node / CLI**: Uses plain `SPARK_APP_ID` env var and in-memory TTL cache. Works with Express, Hono, Fastify, or any CLI tool.
- **Extension**: Hardcodes the public APP_ID directly (no `process.env`). Works for Chrome extensions, browser apps, or Electron.

**Mode:**
- **Poll** (simple): Your client app fetches subscription status on demand. No server-side setup or domain needed.
- **Push**: SparkStripe POSTs signed events to your app's webhook endpoint in real time. Requires a webhook URL configured and tested in the dashboard.

The downloaded zip contains flat files (no subdirectories) that you copy wherever suits your project:

| File | Description |
|------|-------------|
| `SparkStripe-Integration.ts` | Gating helpers, types (`SparkStatus`, `SparkWebhookPayload`, `Tier`, etc.), webhook signature verification (push mode), and a `POST` route handler (push + Next.js). |
| `SparkStripe-Badge.tsx` | React component showing the user's current plan with a status dot and upgrade CTA. Supports `classNames` prop for per-part styling (`root`, `dot`, `label`, `link`). |
| `readme.txt` | Setup instructions and usage examples. |

In push mode for Next.js, create `app/api/webhooks/spark-stripe/route.ts` with a single re-export:
```typescript
export { POST } from '@/lib/SparkStripe-Integration';
```

### Subscription statuses

Every user in SparkStripe has one of these statuses. Use them to gate features in your app:

| Status | Meaning | Grant access? |
|--------|---------|---------------|
| `active` | Recurring subscription is paid and current | Yes |
| `trialing` | Subscription with a Stripe trial period | Yes |
| `lifetime` | One-time payment, never expires | Yes |
| `past_due` | Payment failed, Stripe is retrying (grace period) | Your call |
| `canceled` | Subscription ended or refunded | No |
| `free` | Registered for free trial (no payment yet) | Check `trial.is_expired` |

**Quick access check** (covers all paid plan types):

```javascript
const hasAccess = ['active', 'trialing', 'lifetime'].includes(subscription?.status);
```

### Option A: Poll subscription status

The simplest integration, no server required. Call the status API on demand:

```javascript
const res = await fetch(
  `https://sparkstripe.com/api/public/subscription/status-public?email=${email}&app_id=${APP_ID}`
);
const data = await res.json();
```

Or use the helpers from the gating file:

```typescript
import { getStatus, isPaid, hasAccess, showPaywall, isCanceling, expiresAt } from '@/lib/sparkstripe';

const status = await getStatus(userEmail);
if (isPaid(status))           { /* full access */ }
if (hasAccess(status, 'pro')) { /* pro tier or above */ }
if (showPaywall(status))      { /* trial expired, show upgrade */ }
if (isCanceling(status))      { console.log('Expires:', expiresAt(status)); }
```

**Full response shape:**

```json
{
  "subscription": {
    "status": "active",
    "payment_type": "subscription",
    "price_id": "price_xxx",
    "current_period_end": "2026-04-05T00:00:00.000Z",
    "is_lifetime": false,
    "created_at": "2026-03-05T00:00:00.000Z"
  },
  "verified": true,
  "registered": true,
  "usage_count": 42,
  "trial": null
}
```

For free-tier users (`status: "free"`), `subscription` is `null` and `trial` contains:

```json
{
  "trial": {
    "is_expired": false,
    "days_remaining": 5,
    "uses_remaining": Infinity,
    "expiration_reason": "none",
    "max_days": 7,
    "max_uses": Infinity
  }
}
```

### Option B: Outgoing webhooks

If your app has a server, register a webhook URL and SparkStripe will POST a signed event immediately after every payment lifecycle change.

**Setup:** Dev dashboard, select app, General tab, Edit, paste your endpoint URL (e.g. `https://your-app.com/api/webhooks/spark-stripe`), Save. The `spark_wh_...` secret is shown once on first save, so copy it immediately.

**Events dispatched:**

| Event | Trigger |
|---|---|
| `checkout.completed` | One-time (lifetime) payment succeeded |
| `subscription.activated` | Recurring subscription became active |
| `subscription.trial_started` | Stripe trial period began |
| `subscription.past_due` | Payment failed, Stripe is retrying |
| `subscription.canceled` | Subscription ended or was refunded |

**Payload:**

```json
{
  "event": "subscription.activated",
  "app_id": "my-app",
  "email": "user@example.com",
  "subscription": {
    "status": "active",
    "payment_type": "subscription",
    "price_id": "price_xxx",
    "current_period_end": "2026-04-05T00:00:00.000Z",
    "is_lifetime": false
  },
  "timestamp": "2026-03-05T12:00:00.000Z"
}
```

**`payment_type` values:** `"subscription"` (monthly/yearly recurring) or `"one_time"` (lifetime). When `payment_type` is `"one_time"`, `is_lifetime` is `true` and `current_period_end` is `null`.

**Handler example (Next.js):**

Every webhook request carries a `Spark-Signature` header with the signature `HMAC-SHA256(webhookSecret, rawBody)`. The integration file (push mode) includes `verifyWebhookSignature()` to verify it:

```typescript
// app/api/webhooks/spark-stripe/route.ts
// Option 1: Re-export the built-in handler:
//   export { POST } from '@/lib/SparkStripe-Integration';
// Option 2: Write your own:
import { verifyWebhookSignature, type SparkWebhookPayload } from '@/lib/SparkStripe-Integration';

export async function POST(req: Request) {
  const rawBody = await req.text();
  const sig = req.headers.get('Spark-Signature') ?? '';
  const valid = await verifyWebhookSignature(rawBody, sig);
  if (!valid) return new Response('Unauthorized', { status: 401 });

  const { event, email, data }: SparkWebhookPayload = JSON.parse(rawBody);

  switch (event) {
    case 'checkout.completed':
    case 'subscription.activated':
    case 'subscription.trial_started':
      await db.user.update({
        where: { email },
        data: { isPremium: true },
      });
      break;

    case 'subscription.past_due':
      await db.user.update({
        where: { email },
        data: { paymentWarning: true },
      });
      break;

    case 'subscription.canceled':
      await db.user.update({
        where: { email },
        data: { isPremium: false, paymentWarning: false },
      });
      break;
  }

  return new Response('ok');
}
```

**Important notes:**

- **Idempotency:** Your handler may receive the same event more than once (e.g. network retry). Make sure your logic is safe to re-run.
- **No automatic retries:** SparkStripe dispatches each webhook once with a 10-second timeout. If your endpoint is down, the event is lost. Use [Option A (polling)](#option-a-poll-subscription-status) as a fallback to reconcile missed events on the next user visit.
- **Free trial gating:** Webhooks only fire for Stripe payment events. Free trial expiration is time-based (default: 7 days), so use the polling API to check `trial.is_expired` for free-tier users.

## Testing Payments and Webhooks Locally

`http://localhost:3000/tester` is a built-in integration tester. It shows your subscription status in real time, provides test card details, and lets you switch states instantly.

**1. Start both terminals:**

```bash
# Terminal 1
bun dev

# Terminal 2: forward Stripe events to your local server
stripe listen --forward-to localhost:3000/api/webhooks/stripe
```

Copy the `whsec_...` printed by terminal 2 into `.env.local` as `STRIPE_WEBHOOK_SECRET`, then restart `bun dev`.

**2. Run the flow:**

1. Open `http://localhost:3000/tester` and enter your test email
2. Click the pricing page link and complete checkout with the test card shown
3. Return to the tester: the status card updates live when the webhook fires
4. Use the state switcher buttons to test different subscription states

**Test cards** (any future expiry, any 3-digit CVC):

| Card | Scenario |
|------|----------|
| `4242 4242 4242 4242` | Success |
| `4000 0000 0000 0002` | Declined |
| `4000 0000 0000 9995` | Insufficient funds |
| `4000 0025 0000 3155` | 3D Secure (EU/SCA) |

**Skip checkout** and flip subscription state directly:

```bash
bun run db:switch active your-app-id
bun run db:switch canceled your-app-id
bun run db:switch past_due your-app-id
bun run db:switch lifetime your-app-id
```

> `db:switch` updates the DB only and does not fire outgoing webhooks.

## Dev Dashboard

An admin interface for managing apps, tracking conversions, and testing subscription states. Blocked in production by default via middleware (set `DASHBOARD_ENABLED=true` to enable).

Access is gated by Google OAuth. Only Gmail addresses listed in `DEV_ALLOWED_EMAILS` can log in. If setup is incomplete, a banner with the setup command appears automatically.

**Setup:**
1. Create a Google OAuth app at [console.cloud.google.com](https://console.cloud.google.com/apis/credentials)
   - Redirect URI: `http://localhost:3000/api/auth/callback/google`
2. Add to `.env.local`:
   ```bash
   NEXT_PUBLIC_GOOGLE_CLIENT_ID=xxxxx.apps.googleusercontent.com
   GOOGLE_CLIENT_SECRET=xxxxx
   DEV_ALLOWED_EMAILS=you@gmail.com,colleague@gmail.com
   ```
3. `bun dev`, visit `http://localhost:3000/dashboard/login`

**Features:**
- **Overview**: cross-app analytics (pricing page visits vs actual sales) with daily sparklines, blended conversion rate, and active seasonal sale info
- **Create new apps**: register an app instantly, add pricing plans later via the Plans tab
- **General tab**: app settings, outgoing webhook configuration, test webhook delivery, and integration file download (Next.js, Node/CLI, or browser extension in poll or push mode)
- **Plans tab**: create and delete plans with automatic Stripe Product + Price creation, seed sample plans, sync from Stripe, promote test prices to live
- **Pricing Page tab**: customize comparison tables (glass/opaque styles), seasonal badges, background animations (gradient, bokeh, bubbles, shooting stars), trust badges, and title glow per app
- **Analytics tab**: per-app analytics panel with detailed metrics
- **Follow-up tab**: segment-based email campaigns targeting users nearing expiry, recently expired, or long expired, with per-segment templates and coupon integration
- **Testimonials tab**: manage testimonials and bulk-send one-click testimonial request emails to recent customers
- **FAQ tab**: manage FAQ entries displayed on pricing pages
- **Coupons tab**: create, edit, delete individual coupons, and toggle public listing of discount codes per app
- **Users tab**: inspect users, manually switch subscription states, issue refunds, view user insights (duplicates, near-expiry, past-due, canceling)

## Troubleshooting

**"App not found"**: run `bun run db:list` to verify the app is registered.

**"No pricing plans"**: re-register the app with valid Stripe `price_id` values.

**Webhook not firing**: confirm the endpoint URL matches your deployment and `STRIPE_WEBHOOK_SECRET` is set correctly in Vercel.

<summary><strong>Dev Dashboard: Advanced</strong></summary>

### Page View Tracking

Pricing page visits are tracked automatically with no cookies and no personal data. A fire-and-forget `POST /api/track/page-view` call is made by a tiny client component mounted on every `/pricing/{app_id}` page. Requests are rate-limited to 10 per IP per 5 minutes and fail silently so they never affect visitor UX.

View raw data at `GET /api/dev/analytics?days=30` (dev only, requires dashboard auth).

### Follow-up Email Campaigns

Target users with segment-based email campaigns from the **Follow-up** tab. Three built-in audience segments:

| Segment | Who it targets |
|---------|---------------|
| **Nearing expiry** | Trialing or canceling users within 3 days of expiration |
| **Recently expired** | Canceled less than 30 days ago, or past due |
| **Long expired** | Canceled more than 30 days ago |

Each segment has a customizable email subject and message. Active seasonal badges and coupons are automatically included in campaign emails. Campaigns can be triggered manually from the dashboard or scheduled via QStash cron (`/api/cron/send-segment`).

### Testimonial Requests

From the **Testimonials** tab of any app, pick a time window (customers from the last 1, 3, 6, or 12 months), preview exactly who will receive the email, then hit **Send requests**. Each recipient gets a short personalised email with a mailto CTA pre-filled with the app name, with no forms and no external services.

```
sparkstripe.com
```
